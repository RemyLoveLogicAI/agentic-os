## Overview

Defines a BoardChangeListener type and the KanbanDesk class that wraps a TaskStore to present a safe, renderer-friendly Kanban board API. KanbanDesk maintains change listeners (observer pattern), forwards CRUD and move operations to the underlying TaskStore, and emits notifications after mutating operations so UI layers can re-render.

It also exposes renderer helpers: a read-only columns getter that returns canonical KANBAN_COLUMNS, columnLabel(col) mapping known columns to human-friendly labels with sanitiseString as a fallback, and convenience accessors (getBoard, getTasks, getTask, taskCount). Errors from listeners are caught to avoid breaking the desk.

## Architecture Notes

- Facade + Observer: KanbanDesk is a lightweight facade over TaskStore exposing a simplified API to UI code while implementing an observer pattern (onChange/notify) so renderers can subscribe to board updates.
- Defensive sanitisation: columnLabel uses a static map for known columns and sanitiseString as a fallback, ensuring renderers receive strings and avoid null-reference errors.
- Error isolation: notify() catches and ignores listener exceptions to prevent subscriber failures from breaking the desk or halting notifications.

## Usage Examples

### Main use case

Instantiate KanbanDesk with an existing TaskStore, subscribe to updates with desk.onChange(fn) which returns a deregistration function, then call desk.addTask(...), desk.moveTask(id, 'in_progress'), or desk.removeTask(id). After any mutating call the desk invokes notify() so subscribed UI renderers receive the fresh KanbanBoard from store.snapshot().

## Maintenance Notes

- Listeners: onChange returns a removal function that removes the exact callback reference; callers must keep the reference to unsubscribe. notify() swallows listener exceptions—errors are not propagated.
- TaskStore contract: KanbanDesk relies on TaskStore exposing specific methods (create, move, delete, snapshot, list, get) and a numeric size property—changes in TaskStore API will require updates here.
- Column labels: columnLabel uses a hard-coded labels map for known KanbanColumn values; unknown values are passed through sanitiseString which may modify input for safety.
